# Multi-Agent Architecture Explanation

This document explains the technical architecture behind the Multi-Agent AI Travel Planner.

## High-Level Workflow

The system takes a modular approach, assigning strictly scoped responsibilities to separate LLM instances ("Agents"). This prevents the LLM from becoming confused by trying to solve everything at once.

We utilize **LangGraph** to coordinate these agents. LangGraph treats the workflow as a Directed Acyclic Graph (DAG) state machine, where:
- **State**: The context (inputs and accumulated outputs)
- **Nodes**: Python functions invoking our specific LLMs
- **Edges**: The transition logic connecting the steps

### The Graph Pipeline
1. **START** -> `DestinationAgent`
2. `DestinationAgent` -> `ItineraryAgent`
3. `ItineraryAgent` -> `BudgetAgent`
4. `BudgetAgent` -> **END**

## Detailed Agent Roles

### 1. Destination Agent
- **Input**: User's target location, total days, travel style, and budget tier.
- **Responsibility**: Validate the choice, brainstorm exact cities/regions to visit, and extract the absolute best attractions matching the travel style.
- **Output**: Writes `destinations`, `top_attractions`, and `reasoning` into the global state.

### 2. Itinerary Agent
- **Input**: The chosen `destinations` and `top_attractions` generated by the Destination Agent.
- **Responsibility**: Logically organize attractions across the requested number of days. It ensures no day is overbooked and accounts for the travel style (e.g., placing fewer activities on a "relaxed" day).
- **Output**: Writes `day_wise_plan` into the state.

### 3. Budget Agent
- **Input**: The finalized `day_wise_plan` from the Itinerary Agent, along with the user's selected `budget` tier.
- **Responsibility**: Assign rough USD dollar amounts to different categories (Stay, Food, Transport, Activities) based on the itinerary's density and the location's average cost of living.
- **Output**: Writes `estimated_budget` and `budget_tips` into the state.

## Why This Design is Scalable

1. **Separation of Concerns**: If we want to upgrade the Budget Agent to use a live-pricing API (like Expedia or Google Flights) instead of LLM estimations, we only modify `budget_agent.py`. The rest of the graph remains untouched.
2. **Determinism and State Management**: LangGraph's state object prevents variables from drifting. We define a strict `TypedDict` (`TravelState`), ensuring types never mismatch.
3. **Structured Ouputs**: By relying on LangChain's `with_structured_output` + Pydantic, the LLM is forced to return valid JSON, preventing formatting crashes.
4. **Resiliency**: The try/except blocks in the nodes write fallback data to the state, ensuring that the pipeline continues to the next node even if one agent hits a snag.
